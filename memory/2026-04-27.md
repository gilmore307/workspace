### Core decisions and rules

- All stock/market research timestamps are normalized to U.S. Eastern time / `America/New_York`.
- `task_id`, `run_id`, option event ids, and event standard ids are stable random IDs with semantic prefixes only; do not encode timestamp, contract, or trigger semantics in ids.
- Final saved flat row outputs are CSV-only. JSONL can exist only as transient/run-local cleaned data. Nested point-in-time/context outputs may be final JSON.
- High-volume provider rows are transient by default. Persist final cleaned/aggregated outputs only.
- Bundles are `kind=data_bundle`; final saved/used data categories are `kind=data_kind`.
- Data kind templates are source-organized under `trading-data/templates/data_kinds/<source>/` and should describe final output, not raw provider dumps.
- Provider `term.path` stores public documentation URL. Source-secret `config.path` can mirror local source secret JSON path, but secret values stay outside Git.
- Macro acquisition uses one parameterized `macro_data` bundle; source/dataset/release/series are task params.
- FRED is limited to FRED/St. Louis Fed/ALFRED-unique or explicitly approved FRED-native data. Official agency measures should come from official sources by default.
- OKX is canonical for crypto execution research. Alpaca crypto may be cross-check only.
- Alpaca/OKX market-facing schemas should share common field names where semantics match. Consumers must tolerate partial feature availability.
- ThetaData options are split by use case, not endpoint family:
  - `thetadata_option_selection_snapshot`
  - `thetadata_option_primary_tracking`
  - `thetadata_option_event_timeline`
- ThetaData `option_chain_snapshot` stores complete visible/unexpired chain; no filtering/selection in the data pipeline. Contract selection belongs to future model work.
- `option_activity_event` is news-like and uses fields: `data_kind,id,headline,created_at_et,updated_at_et,symbols,summary,url`.
- `option_activity_event.url` is `<id>.json`.
- `option_activity_event.summary` lists only abnormal indicator type names (e.g. `trade_at_ask;opening_activity;iv_high_cross_section`). No normal metrics, trigger values, current standards, or scoring.
- `option_activity_event_detail.triggered_indicators` is an object keyed by abnormal indicator type.
- `option_activity_event_detail` carries objective `statistics`, event-time `current_standard`, and `standard_context` (`standard_source`, `standard_id`, `standard_generated_at_et`). `current_standard` is model/run-specific and not a global fixed threshold.
- Model standard identity/versioning is deferred to `trading-model`.
- `timeframe` must be an input parameter for all bar/liquidity-like outputs.
- Registry `kind=field` is a generic contract/schema slot and is not automatically a current table-column list.
- Field naming rule established: use the broadest truthful field key. Reusable metrics/columns use generic keys (`BAR_CLOSE`, `QUOTE_BID`, `TRADE_SIZE`, `GREEK_DELTA`). Asset-class keys are allowed when inherently asset-class-specific. Scenario prefixes stay only for scenario-specific meanings (`OPTION_EVENT_DETAIL_PRICE_VS_ASK`). Do not keep implementation/template-era prefixes like `OPTION_TEMPLATE_*` for reusable current data fields.
- Do not merge semantically different volume/size grains:
  - `trade_size` = single triggering option trade print size.
  - `window_volume` = option event evidence-window aggregate.
  - `trade_volume` = liquidity interval aggregate.
  - `volume` = OHLCV bar volume.
